\subsection{Gantt Chart}

The Gantt chart visually represents the timing and overlap of major activities such as requirement analysis, sprint planning, interface design, module development, testing, and final review \cite{gantt1}. In this revised plan, the chart is aligned with the eight sprint structure defined in the Agile process model so that the implementation schedule and the written process model remain consistent.

The chart should be read together with the sprint descriptions because each four-week sprint represents a focused development increment with specific modules and review outcomes. When viewed in that way, the Gantt chart becomes a practical coordination tool rather than only a visual schedule \cite{spp1}.

\subsection{Work Breakdown Structure}

The work breakdown structure decomposes the project into manageable work packages \cite{wbs1}. It helps the team see how the final system emerges from smaller responsibilities such as requirement analysis, dashboard planning, scheduling logic, monitoring logic, submission handling, reporting, testing, and documentation.

\subsection{Critical Path Method}

The critical path method identifies the chain of activities that most directly determines whether the project can finish on time \cite{spp1}. In this FYP, activities such as requirement stability, design completion, core-module development, integration, testing, and final documentation are part of that path. The critical path perspective is valuable because it helps the team recognize that some delays are more dangerous than others.

\subsubsection{Schedule Control and Review}

Schedule control in this project depends on regular sprint review, comparison of completed tasks against planned outcomes, and early adjustment when a task begins to slip \cite{scrum1}. This approach is more useful than passive calendar tracking because it links time management with actual deliverables such as working modules, diagrams, documentation updates, and testing evidence.
